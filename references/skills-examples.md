# Skill Examples

Skills are Markdown files that teach Claude reusable workflows. They live in `.claude/skills/<name>/SKILL.md` (project-specific) or `~/.claude/skills/<name>/SKILL.md` (personal, cross-project).

---

## Example 1: Commit Skill

**Location**: `.claude/skills/commit/SKILL.md`

```markdown
---
name: commit
description: This skill should be used when the user asks to "commit", "create a commit", "commit changes", or "save my work".
version: 1.0.0
---

When creating a commit, follow this process:

1. Run `git status` and `git diff --staged` to understand what's being committed
2. If nothing is staged, ask the user what to stage
3. Write a commit message following this format:
   - First line: type(scope): description (max 72 chars)
   - Types: feat, fix, refactor, docs, test, chore, style
   - Blank line
   - Body: explain WHY, not WHAT (the diff shows what)
   - NEVER include AI attribution (no Co-Authored-By, no "Generated by")
4. Never use --no-verify
5. Never amend unless explicitly asked
6. Show the commit hash and message when done
```

## Example 2: Deploy Skill

**Location**: `.claude/skills/deploy/SKILL.md`

```markdown
---
name: deploy
description: This skill should be used when the user asks to "deploy", "push to production", or "release".
disable-model-invocation: true
version: 1.0.0
---

IMPORTANT: This skill has side effects. Only run when the user explicitly invokes /deploy.

Deployment checklist:

1. Verify the current branch is `main` or `master`
2. Run the test suite — abort if any test fails
3. Run the build — abort if build fails
4. Check for uncommitted changes — abort if dirty
5. Show the user what will be deployed:
   - Current branch and latest commit
   - Diff since last deployment tag (if tags are used)
6. Ask for final confirmation before proceeding
7. Run the deploy command: [REPLACE WITH YOUR DEPLOY COMMAND]
8. Verify the deployment succeeded
9. Create a git tag with the deployment date: deploy-YYYY-MM-DD-HHMMSS
```

## Example 3: Test Runner Skill

**Location**: `.claude/skills/test/SKILL.md`

```markdown
---
name: test
description: This skill should be used when the user asks to "run tests", "test this", "check if tests pass", or after making code changes that need verification.
version: 1.0.0
---

When running tests:

1. Determine the scope:
   - If the user specified files/tests, run only those
   - If recent edits were made, run tests related to the changed files
   - Otherwise, run the full suite

2. Run the test command:
   - [REPLACE: e.g., `npm test`, `pytest`, `php artisan test`]

3. Analyze results:
   - If all pass: report the count and confirm success
   - If failures: analyze each failure, determine if it's a real bug or a test issue
   - If the failure is from code you just wrote, fix it and re-run

4. Never skip or ignore failing tests
5. Never modify tests just to make them pass (unless the test itself is wrong)
```

## Example 4: Code Review Skill

**Location**: `~/.claude/skills/review/SKILL.md` (personal, cross-project)

```markdown
---
name: review
description: This skill should be used when the user asks to "review this code", "check my changes", "review the PR", or "code review".
version: 1.0.0
---

When reviewing code:

1. Get the diff: `git diff` for unstaged, `git diff --staged` for staged, or `git diff main...HEAD` for branch
2. For each changed file, check:
   - Logic errors and edge cases
   - Security issues (injection, XSS, credential exposure)
   - Performance concerns (N+1 queries, unnecessary loops)
   - Missing error handling
   - Breaking changes to public APIs
3. Do NOT flag:
   - Style preferences (that's the linter's job)
   - Minor naming opinions
   - Missing comments on self-explanatory code
4. Report findings as:
   - **Critical**: Must fix before merge (bugs, security)
   - **Suggestion**: Would improve the code but not blocking
5. If no issues found, say so clearly — don't invent problems
```

## Example 5: Figma Component Decomposition Skill

**Location**: `.claude/skills/figma-component/SKILL.md`

For design-driven projects where Figma designs are converted to code. This skill decomposes a page-level Figma design into individual components and implements each one, with duplicate detection to prevent recreating existing components.

```markdown
---
name: figma-component
description: Break down a Figma page design into individual components and implement each one. Use when the user provides a Figma URL and wants component-level implementation.
version: 1.0.0
---

When decomposing a Figma design into components:

1. Get page metadata (`get_metadata`) to see the full component tree
2. Get page screenshot for visual reference
3. List all distinct components found and present to the user
4. Ask which components to implement

For EACH component, before writing any code:

5. **Duplicate detection (REQUIRED):**
   - Search existing component directories for similar files
   - Search CSS for matching class names
   - If a potential match is found, compare:
     - Same data fields? Same layout? Same purpose?
     - Take a screenshot of the existing component and compare visually
   - Report: "exact match" / "needs updates" / "similar but different variant" / "new component"
   - Wait for user confirmation

6. **Handle variants:**
   - If Figma shows multiple variants (e.g., card with/without image), list them all
   - Ask the user which to implement — never silently pick one

7. **Map to data layer:**
   - What data does this component need?
   - Do the fields/models already exist?
   - Ask before creating new data structures

8. **Implement:** Create/update template, styles, and interactivity
   - Follow project conventions (naming, CSS methodology, etc.)
   - Colors via design tokens/CSS variables, never hardcoded
```

This pattern solves the most common Figma-to-code problems:
- **Recreating existing components** — duplicate detection catches this
- **Mixing up variants** — explicit variant handling prevents confusion
- **Inventing data structures** — mapping to existing data layer first

## Example 6: Tasks Skill (Combo Pattern)

**Location**: `.claude/skills/tasks/SKILL.md`

This skill works **in tandem with a task-manager agent** (see Example 4 in agents-examples.md). The skill handles user interaction and lightweight reads; the agent handles multi-file sync operations in isolated context.

```markdown
---
name: tasks
description: Manage project tasks. Use when the user says "tasks", "what's next", "add task", "mark done", "update status", "show tasks", or "sync status". Reads and updates docs/todo.md, plan docs, and STATUS.md.
version: 1.0.0
---

When invoked:

1. **Read current state**
   - Read `docs/todo.md` to understand current task status
   - Summarize: in-progress items, next pending items, any blockers

2. **Determine action from user input**
   - `/tasks` (no args) → Show status summary
   - `/tasks add "description"` → Add a new task to docs/todo.md
   - `/tasks done <item>` → Mark a task as complete
   - `/tasks next` → Show the next actionable task
   - `/tasks sync` → Sync status across all tracking files

3. **Handle simple actions inline**
   - Adding a single task: edit docs/todo.md directly
   - Marking done: edit docs/todo.md directly
   - Showing status: read and summarize

4. **Delegate complex actions to the task-manager agent**
   - Syncing across plan docs: invoke the task-manager agent
   - Bulk updates: invoke the task-manager agent
   - Pass context: current task state, what changed, which files to update

5. **Report back**
   - Summarize what changed
   - Show updated task counts (done/total)
   - Highlight any newly unblocked tasks
```

This combo pattern keeps the main conversation context clean — the skill does quick reads and simple edits, while the agent handles multi-file operations without polluting the context window. See `templates/agent-skill-combo/` for reusable templates.

## Key Decisions When Creating Skills

| Question | Guidance |
|----------|----------|
| Project or personal? | If it's project-specific (deploy commands, test setup), put it in `.claude/skills/`. If it applies everywhere (commit format, review process), put it in `~/.claude/skills/`. |
| User-invocable? | Default is yes (appears as /skill-name). Set `user-invocable: false` if it should only be auto-triggered. |
| Disable model invocation? | Set `disable-model-invocation: true` for anything with side effects (deploy, publish, send messages). |
| How long? | Keep SKILL.md under 500 lines. Use supporting files in the same directory for reference material. |
| Combo pattern? | If the workflow needs both a user command AND isolated execution (e.g., task management, multi-file sync), pair a skill with an agent. The skill handles UX; the agent handles the work. See `templates/agent-skill-combo/`. |
